# AP Question Bank - 全局扩展性架构调研报告

> 商业目标：通过开源免费题库获取流量，推广翎英教育业务
> 
> 核心要求：架构必须支持未来扩展（营销页面、在线报名、科目介绍、更多科目），避免大规模返工
> 
> 调研维度：商业架构、平台分层、路由预留、组件复用、内容扩展、数据模型

---

## 一、商业目标与架构映射

### 1.1 你的商业模式

```
流量漏斗：
  免费题库（吸引搜索流量）→ 科目介绍/内容（建立信任）→ 业务介绍（转化）→ 在线报名（成交）
```

这意味着网站需要同时承载三种类型的页面：

| 页面类型 | 例子 | 目的 | 优先级 |
|---------|------|------|--------|
| **功能页** | Quiz、Mock、错题本 | 提供价值，留住用户 | P0（现在） |
| **内容页** | 科目介绍、考试介绍、备考指南 | SEO 流量，建立专业形象 | P1（近期） |
| **营销页** | 翎英教育介绍、课程展示、在线报名 | 商业转化 | P2（未来） |

### 1.2 架构必须回答的问题

1. **新页面加在哪里？** 不破坏现有功能
2. **新科目怎么加？** 不修改现有代码
3. **营销内容怎么放？** 不影响用户体验
4. **导航怎么组织？** 功能页、内容页、营销页不混在一起

## 二、竞品架构深度调研

### 2.1 Khan Academy（教育平台标杆）

```
架构分层：
  平台层：首页、关于我们、捐赠、 Careers
  科目层：数学、科学、经济、艺术...（每个科目有独立 Landing Page）
  内容层：课程视频、练习题、单元测试
  功能层：进度追踪、成就系统、讨论区
```

**关键设计：**
- 每个科目有独立的 Landing Page（`/math`、`/economics`）
- 科目页面内部有子导航：课程、练习、单元
- 平台级导航始终可见，科目级导航在科目页面内
- 营销内容（捐赠、About）在平台层，不干扰学习

**对我们的启示：**
- 首页不应直接是功能页，而是"科目选择 + 平台介绍"的组合
- 每个科目需要独立的 Landing Page（科目介绍、考试介绍）
- 功能页（Quiz/Mock）应该在科目上下文内，但路由要清晰

### 2.2 StudySmarter（现 Vaia，欧洲教育 App）

```
架构：首页 → 选择学习集 → 学习模式（闪卡/测试/匹配）
关键设计：
  - 首页展示所有学习集（类似 Anki 的 Deck 列表）
  - 每个学习集有详情页（介绍、统计）
  - 学习模式独立于学习集，切换学习集自动切换内容
```

**对我们的启示：**
- 首页=科目卡片列表，每个卡片显示进度
- 点击科目后进入"科目主页"，展示该科目的功能入口
- 功能页保持独立，通过 Context 获取当前科目

### 2.3 Duolingo（语言学习，多课程切换）

```
架构：首页（学习路径）+ 顶部旗帜切换（课程）+ 底部导航（功能）
关键设计：
  - 顶部旗帜：切换课程，所有内容自动刷新
  - 底部导航：主页、排行榜、任务、商店、个人资料
  - 课程之间数据完全隔离，但 UI 框架复用
```

**对我们的启示：**
- 科目切换器放在顶部，像 Duolingo 的旗帜一样醒目
- 切换科目后，当前页面数据自动刷新，不跳转
- 底部/侧边导航可以预留扩展位（比如"课程""关于"）

### 2.4 国内竞品：猿辅导、作业帮

```
架构特点：
  - 首页：品牌展示 + 课程推荐 + 功能入口
  - 课程页：科目介绍、课程大纲、老师介绍、报名按钮
  - 功能页：题库、练习、错题（需要登录）
  - 营销内容穿插在内容页中（如"点击领取免费试听课"）
```

**对我们的启示：**
- 题库是流量入口，但必须在科目页中自然引流到业务介绍
- 每个科目可以有一个"科目主页"，包含：考试介绍 + 功能入口 + 课程推荐
- 报名系统可以作为独立页面，也可以作为嵌入式按钮

## 三、推荐架构："平台-科目-功能" 三层模型

### 3.1 架构总览

```
┌─────────────────────────────────────────────┐
│              平台层（Platform）              │
│  首页 / 品牌介绍 / 关于我们 / 在线报名 / 设置  │
└─────────────────────────────────────────────┘
                     ↓
┌─────────────────────────────────────────────┐
│              科目层（Subject）               │
│  科目主页 / 科目介绍 / 考试介绍 / 备考指南    │
│  ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐          │
│  │Macro│ │Micro│ │Stats│ │ ... │          │
│  └─────┘ └─────┘ └─────┘ └─────┘          │
└─────────────────────────────────────────────┘
                     ↓
┌─────────────────────────────────────────────┐
│              功能层（Feature）               │
│  Quiz / Mock / 搜索 / 错题 / 记录 / 历史    │
│  （每个功能页通过 Context 读取当前科目）      │
└─────────────────────────────────────────────┘
```

### 3.2 路由设计（预留扩展）

```
当前路由（不变）：
  /                  → 首页（科目卡片列表）
  /quiz              → Quiz 设置
  /play              → Quiz 做题
  /exam              → Mock 设置
  /frq               → FRQ 做题
  /frq-score         → FRQ 评分
  /score             → 分数统计
  /search            → 搜索
  /mistakes          → 错题本
  /history           → 记录

预留路由（未来扩展）：
  /about             → 翎英教育介绍（营销页）
  /courses           → 课程列表（营销页）
  /enroll            → 在线报名（营销页）
  /subject/:id       → 科目主页（内容页）
  /subject/:id/guide → 考试介绍/备考指南（内容页）
  /subject/:id/syllabus → 课程大纲（内容页）
  /settings          → 用户设置（已预留）
```

**关键原则：**
- 现有功能路由不变，兼容老用户书签
- 新路由用 `/subject/:id/` 前缀，与功能路由区分
- 营销路由用 `/about`、`/courses` 等短路径，利于 SEO

### 3.3 页面类型分类

| 类型 | 路由模式 | 导航栏特点 | 例子 |
|------|---------|-----------|------|
| **平台页** | `/` `/about` `/enroll` | 显示全站导航 | 首页、品牌介绍 |
| **科目页** | `/subject/:id` | 显示科目子导航 | 科目主页、考试介绍 |
| **功能页** | `/quiz` `/exam` | 显示科目切换器 + 功能导航 | Quiz、Mock |
| **营销页** | `/courses` `/enroll` | 可能有不同的 CTA | 在线报名 |

## 四、导航栏设计（全局组件）

### 4.1 当前导航栏的问题

```
当前：首页 | Quiz | Mock | 搜索 | 错题 | 记录
问题：
  1. 没有科目切换器
  2. 没有空间放营销入口（关于我们、课程）
  3. 移动端菜单会很长
```

### 4.2 推荐导航栏设计

```
桌面端：
  [翎英教育 Logo]    [科目切换器▼]  [Quiz] [Mock] [搜索] [错题] [记录]    [关于我们] [课程]

移动端：
  [翎英教育 Logo]    [科目切换器▼]  [≡]
  
  ≡ 菜单展开：
    - Quiz
    - Mock
    - 搜索
    - 错题
    - 记录
    ──────────
    - 关于我们（营销）
    - 课程（营销）
```

**设计要点：**
- 科目切换器始终可见，像 Duolingo 的旗帜
- 功能导航在左侧，营销入口在右侧，视觉上分离
- 移动端用折叠菜单，营销入口放底部

### 4.3 科目切换器 UX

```
切换器样式：
  [📚 Macro ▼]
  
下拉菜单：
  📚 AP Macroeconomics      [当前]
  📚 AP Microeconomics
  📊 AP Statistics          [未来]
  ────────────────────
  + 添加/管理科目
```

**行为：**
- 切换科目后，当前页面数据自动刷新（不跳转）
- 如果用户在科目主页（`/subject/:id`），切换后跳转到新科目主页
- 如果用户在功能页（`/quiz`），切换后只刷新内容

## 九、实施路线图（分阶段）

### 阶段 1：多科目基础（现在）

目标：支持 Macro + Micro，不影响现有功能

- 数据层：合并科目数据，JSON 内部加 `subject` 字段
- 存储层：科目前缀 + 自动迁移
- 状态层：`SubjectContext` 管理当前科目
- UI 层：导航栏加科目切换器，功能页接入 Context
- 首页：显示科目卡片（Macro + Micro）

### 阶段 2：内容页扩展（近期）

目标：每个科目有独立的 Landing Page 和考试介绍

- 新增路由：`/subject/:id`、`/subject/:id/guide`
- 新增组件：`ContentPage`、`SubjectLandingPage`
- 新增数据：Markdown 格式的考试介绍、备考指南
- 首页：科目卡片增加"科目主页"入口
- 科目主页：考试介绍 + 功能入口 + 预留课程推荐位

### 阶段 3：营销页扩展（未来）

目标：品牌展示、业务介绍、在线报名

- 新增路由：`/about`、`/courses`、`/enroll`
- 新增组件：`MarketingPage`、`CourseCard`、`EnrollForm`
- 导航栏：增加"关于我们""课程"入口
- 科目主页：嵌入课程推荐和报名按钮
- 功能页：顶部预留可关闭的营销横幅位

### 阶段 4：功能扩展（未来）

目标：新增功能（闪卡、笔记、讨论区等）

- 新增功能：在 `FEATURE_REGISTRY` 中注册
- 导航栏自动渲染新功能入口
- 数据存储：新增 `macro_flashcards`、`macro_notes` 等 key
- 不影响现有功能

## 十、为什么这个架构不会返工

| 设计决策 | 防返工机制 |
|---------|-----------|
| **路由分层** | 平台路由 `/about`、科目路由 `/subject/:id`、功能路由 `/quiz` 互不干扰，新增页面不会破坏现有功能 |
| **Context 管理科目** | 功能页通过 Context 读取当前科目，不依赖 URL 参数，路由可以不变 |
| **功能注册表** | 新增功能只需注册，导航栏自动渲染，不用改 Header 代码 |
| **组件预留接口** | `SubjectCard`、`ContentPage` 提前设计接口，未来填充内容即可 |
| **数据科目前缀** | 新增科目只需增加数据文件 + 配置，存储层自动隔离 |
| **营销位预留** | 功能页顶部预留 `MarketingBanner` 位，现在关闭，未来打开即可 |
| **GitHub Pages 兼容** | 路由不改、base 不改、HashRouter 不变，部署不会出问题 |

## 十一、调研结论

1. **三层架构**：平台层（营销）+ 科目层（内容）+ 功能层（题库），每层独立扩展
2. **路由预留**：现有功能路由不变，新增科目用 `/subject/:id/`，营销用 `/about`、`/enroll`
3. **Context 驱动**：功能页通过 Context 获取当前科目，不反映在 URL 中，路由保持简洁
4. **组件复用**：`SubjectSwitcher`、`SubjectCard`、`ContentPage` 等全局组件提前设计
5. **首页重构**：首页是科目卡片列表 + 品牌 Hero + 营销位，不是直接进入功能
